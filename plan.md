# Genome Editor: Tauri Hybrid Project Plan

## Background

Open Vector Editor（OVE）は React+Redux+BlueprintJS ベースの DNA 配列エディタですが、大規模配列での性能問題が深刻です。

### OVE の性能ボトルネック

| ボトルネック | 原因 |
|---|---|
| SVG DOM ノードの爆発 | 大配列でアノテーション・cutsite・ラベルが大量生成 |
| Redux selector の再計算 | 制限酵素サイト計算が O(n×m) で毎回再実行 |
| React 再レンダリング | 状態変更で 80〜100 個のコンポーネントが連鎖的に再描画 |
| バンドルサイズ | UMD 全体で約 46.6MB、48 の依存パッケージ |

プロジェクト規模: コアソースコード約 53,700 行、316 JS ファイル、80〜100 React コンポーネントです。

### 代替ツール調査結果

フルエディタ機能を持つ軽量オープンソースの代替は、現時点で OVE 以外に存在しません。

- SeqViz: MIT ライセンスのビューア（編集機能なし）。Ginkgo Bioworks が採用
- ORFView: Tauri+React+SeqViz ベースのデスクトップアプリ（ビューアが主目的）
- PlasCAD: Rust/egui 製だが機能が限定的（v0.7.9）
- UGENE: C++/Qt 製で成熟しているが、組み込み用途には不向き

### アプローチ比較

| アプローチ | 工数 | リスク | パフォーマンス改善 |
|---|---|---|---|
| A. 完全 Rust GUI 移植 | 16〜25 人月（+50% リスク） | 極めて高い | 最大 |
| B. Tauri ハイブリッド（採用） | 6〜10 人月 | 中 | 大 |
| C. WASM 部分移植 | 3〜5 人月 | 低 | 中（計算処理のみ） |

### 期待される改善

| 指標 | Electron/ブラウザ | Tauri ハイブリッド |
|---|---|---|
| メモリ使用量 | 200〜300 MB | 30〜40 MB |
| 起動時間 | 1〜2 秒 | 0.5 秒未満 |
| バンドルサイズ | 100 MB+ | 10 MB 未満 |
| 制限酵素計算 | ベースライン | 10〜100 倍高速 |

## Phase 0: SeqViz PoC（1〜2 週間）

### 目的

ビューア要件の検証です。SeqViz がプロジェクトの基盤として十分かを判断します。

### マイルストーン

- M0.1: SeqViz で GenBank ファイルを読み込み、円形/線形ビューを表示
- M0.2: アノテーション表示、制限酵素サイト表示の動作確認
- M0.3: 大規模配列（10kb+）での描画パフォーマンス計測
- M0.4: Go/No-Go 判定レポート作成

### 判定基準

- 円形/線形ビューの描画が 1 秒以内
- 10kb 配列でスクロールが 60fps を維持
- 必要な表示要素（アノテーション、ORF、制限酵素サイト）がカバーされている

## Phase 1: 計算ロジックの Rust/WASM 移植（2〜3 人月）

### 目的

パフォーマンスボトルネックの根本的な解決です。CPU 集約型の計算処理を Rust に移行し、WASM 経由でフロントエンドから呼び出します。

### マイルストーン

- M1.1: Rust ワークスペース構築、CI/CD パイプライン整備
- M1.2: GenBank/FASTA パーサー実装（gb-io ベース）
- M1.3: 制限酵素データベース＋検索エンジン実装
- M1.4: ORF 検出アルゴリズム実装
- M1.5: WASM ビルド＋React 統合テスト
- M1.6: ベンチマーク計測、JS 実装との比較レポート

### 成功基準

- 制限酵素計算が JS 版の 10 倍以上高速
- GenBank パースが JS 版の 5 倍以上高速
- WASM モジュールサイズが 2MB 以下

## Phase 2: Tauri 化（1〜2 人月）

### 目的

Electron からの脱却です。Tauri v2 でデスクトップアプリ化し、メモリ使用量とバンドルサイズを削減します。

### マイルストーン

- M2.1: Tauri v2 プロジェクトスキャフォールド
- M2.2: Phase 1 の Rust モジュールを Tauri コマンドとして統合
- M2.3: ファイル I/O（ネイティブダイアログ経由の GenBank/FASTA 読み書き）
- M2.4: ローカルファイル監視（ホットリロード対応）
- M2.5: クロスプラットフォームビルド（macOS / Windows / Linux）
- M2.6: インストーラー・自動アップデート機能

### 成功基準

- メモリ使用量が 50MB 以下
- 起動時間が 1 秒未満
- バンドルサイズが 15MB 以下

## Phase 3: フロントエンド最適化（必要に応じて）

### 目的

Phase 1-2 で解決しきれなかった描画パフォーマンスの問題を解消します。

### 候補施策

- SVG から Canvas/WebGL への段階的移行
- 仮想スクロールの導入
- Web Worker を利用した非同期レンダリング
- React コンポーネントのメモ化最適化

### 判定基準

Phase 2 完了時のベンチマーク結果に基づき、必要な施策を選定します。

## リスク管理

| リスク | 影響度 | 対策 |
|---|---|---|
| SeqViz のカスタマイズ性が不足 | 高 | Phase 0 で早期検証、必要なら Fork |
| WASM-JS 間のデータ転送オーバーヘッド | 中 | SharedArrayBuffer/Zero-copy 転送を検討 |
| Tauri v2 の macOS 署名・公証 | 中 | Apple Developer Program 早期取得 |
| OVE のエディタロジックが密結合 | 中 | 段階的に切り出し、既存 UI を維持 |
| rust-bio の API 安定性 | 低 | バージョン固定、ラッパー層で吸収 |
